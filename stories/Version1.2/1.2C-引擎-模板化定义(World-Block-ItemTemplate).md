# **研发日志：关于“模板”（Template）的设计思路**

大家好，  
接续上一篇关于统一注册表（Registry）的日志，今天我想谈谈我们放入注册表中的那些“内容”本身的设计思路。  
我们准备将游戏中所有核心内容的“定义”都统一命名为 ...Template（模板）。例如，我们已经有了 BlockTemplate（方块模板）和 WorldTemplate（世界模板），未来还会有 ItemTemplate（物品模板）等等。

### **面向开发者（Developer）**

这个命名约定背后，是一种清晰的设计分离：**定义与实例的分离**。

* **...Template (模板/定义):** 这是一个静态的、不变的“蓝图”。  
  * BlockTemplate 定义了一个方块*是*什么——它的硬度、挖掘等级、它有哪些状态（比如朝向）。  
  * WorldTemplate 定义了一个世界*是*什么——它的天空颜色、重力、高度限制。

这就像是原版Minecraft中的 Block 或 DimensionType 对象。它们本身是注册表中的单例，提供了所有“原型”属性和行为。

* **...Instance (实例):** 这是“蓝图”在游戏世界中的具体体现。  
  * 一个 BlockState（方块状态）在坐标 (10, 20, 30)，就是 BlockTemplate 的一个实例。  
  * 一个实际运行的 SharedWorld（游戏世界），就是 WorldTemplate 的一个实例。

通过这种方式，我们为所有类型的游戏内容（方块、物品、世界）建立了一个统一的抽象模式。

### **面向用户（User）**

这个“模板”概念对您来说意味着什么？  
简单来说，它确保了游戏世界的**一致性**和**可扩展性**。

* **一致性：** BlockTemplate 就像是“橡木”的DNA。它规定了所有“橡木”方块都长什么样、有多硬、能不能被点燃。无论您在世界中放置多少个橡木方块，它们都遵循这套相同的规则，因为它们共享同一个“模板”。  
* **可扩展性：** 当我们想要添加一种全新的方块时，我们只需要定义一个新的 BlockTemplate（比如“枫木”），然后将它注册到系统中。游戏就会自动知道如何处理它。

这种将“规则”（Template）和“实体”（Instance）分开的做法，是构建一个复杂、稳定且易于扩展的游戏世界的基础。
